# 4. 아키텍처

# 4.1.1 MySQL 엔진 아키텍처

- MYSQL 서버는 다른 DBMS에 비해 구조가 상당히 독특하다
  - 이런 이유 때문에 다른 DBMS에서는 가질 수 없는 이점도 있고, 단점도 가지고 있다.
- MySQL은 대부분의 프로그래밍 언어로부터 접근 방법을 모두 지원하고, **MySQL은 MySQL 엔진과 스토리지 엔진으로 구분할 수 있다.**

# 1. MySQL 엔진

- **커넥션 핸들러** : 클라이언트가 전달하는 접속 요청 및 쿼리 요청을 처리하는 역할을 한다.
- **SQL 파서** : MySQL 서버에서 실행되는 쿼리를 분석하고 구문 분석하는 역할을 한다.
- **SQL 인터페이스**: 사용자가 MySQL 데이터베이스의 상호 작용할 수 있는 방법을 제공하는 역할을 한다.
- **옵티마이저**: MySQL 엔진에서 실행될 쿼리의 최적 실행 계획을 결정하는 역할을 한다.
- **캐시**: MySQL이 자주 액세스하는 데이터나 인덱스를 메모리에 저장해서 쿼리 성능을 향상 시키는 역할을 함.
- **버퍼**: MySQL 서버의 디스크 액세스를 줄이기 위해 사용, 디스크 I/O 액세스를 최소화 → DB 성능을 향상 시키는 역할을 함.

### 결론적으로 MySQL 엔진은 SQL 문장을 분석하거나 최적화하는 등 DBMS의 두뇌에 해당하는 처리를 수행한다.

# 2. 스토리지 엔진

- 데이터를 저장하고 검색하는 데 사용되는 엔진이다.
- MySQL 엔진은 하나지만, 스토리지 엔진은 여러 개를 동시에 사용 가능함.
- 아래와 같이 테이블이 사용할 스토리지 엔진을 지정하면 이후 해당 해당 테이블의 모든 작업들은 스토리지 엔진이 처리함.

```sql
mysql> CREATE TABLE test_table (fd1 INT, fd2 INT) ENGINE=INNODB;
```

- 각각의 스토리지 엔진은 성능 향상을 위해 다양한 기능을 제공
  - ex) MyISAM 스토리지 엔진의 키 캐시, InnoDB의 InnoDB 버퍼 풀

### 결론적으로 스토리지 엔진은 실제 데이터를 디스크 스토리지에 저장, 읽어오는 부분을 전담한다.

# 3. 핸들러 API

- MySQL 엔진에서 스토리지 엔진에 **데이터의 변경과 탐색과 관련된 요청**을 하는데, 이러한 요청을 **핸들러 요청**이라 한다. 그리고 이 때 사용되는 API를 핸들러 API라 한다.
- MySQL 엔진은 이 핸들러 요청들을 통해서만 디스크로부터 데이터를 읽고, 쓸 수 있음.
- 핸들러 API를 통해 얼마나 데이터를 읽고 쓰기 위해 디스크에 왔다 갔다 했는지 분석 할 수 있다.

# 4.1.2 MySQL 스레딩 구조

- MySQL 서버는 프로세스 기반이 아니라 스레드 기반으로 작동함.
- 스레드는 포그라운드(Foreground) 스레드와 백그라운드(Background) 스레드로 구분할 수 있다.

```sql
SELECT thread_id, name, type, processlist_user, processlist_host
FROM performance_schema, threads ORDER BY type, thread_id;
```

- 전체 44개의 스레드가 실행 중이며, 그 중에서 41개의 스레드가 백그라운드 스레드, 3개가 포그라운드 스레드이다.

# 4.1.2.1 포그라운드 스레드(클라이언트 스레드)

- 포그라운드 스레드는 최소한 MySQL 서버에 접속된 클라이언트의 수만큼 존재한다.
- 하는 역할은 주로 각 클라이언트 사용자가 요청하는 쿼리 문장을 처리한다.
- 클라이언트가 커넥션을 종료하면, 해당 커넥션을 담당하는 스레드는 다시 스레드 캐시로 돌아간다.
- 스레드 캐시에 일정 갯수 이상의 스레드가 존재한다면, 되돌아오는 스레드를 그냥 종료시킨다.

### 작업 방식

1. 데이터를 MySQL의 데이터 버퍼나 캐시로부터 가져온다.
2. 버퍼나 캐시에 없는 경우→ 직접 디스크의 데이터나 인덱스 파일로부터 데이터를 읽어와서 작업을 처리.

# 4.1.2.2 백그라운드 스레드

- InnoDB는 다음과 같이 여러가지 작업을 백그라운드로 처리한다.
  - 인서트 버퍼를 병합하는 스레드
  - 로그를 디스크로 기록하는 스레드
  - InnoDB 버퍼 풀의 데이터를 디스크에 기록하는 스레드
  - 데이터를 버퍼로 읽어 오는 스레드
  - 잠금이나 데드락을 모니터링하는 스레드
- 이 중 가장 중요한 스레드는  로그 스레드(Log thread)와 쓰기 스레드(Write thread)일 것이다.
- **읽기 스레드는 포그라운드 스레드가 담당해 많이 설정할 필요 X**, **쓰기 스레드는 아주 많은 작업을 백그라운드**로 처리하기 때문에 **내장 디스크를 사용할 때는 2~4**, **DAS SAN과 같은 스토리지를 사용할 때는 충분히 설정하는 것이 좋다.**

# 4.1.3 메모리 할당 및 사용 구조

MySQL에서 사용하는 메모리 공간은 **글로벌 메모리 영역**과 **로컬 메모리 영역**으로 구분함.

- **글로벌 메모리 영역**은 모든 공간을 MySQL 서버가 시작하면 **OS로부터 할당**된다.

# 4.1.3.1 글로벌 메모리 영역

일반적으로 클라이언트 스레드의 수와 무관하게 **하나의 메모리 공간만 할당**

단, 필요에 따라 **2개 이상의 공간을 할당받을 수도 있지만 스레드 수와 무관**하고 **모든 스레드에 의해 공유**됨.

- 대표적인 글로벌 메모리 영역
  - 테이블 캐시
  - InnoDB 버퍼 풀
  - InnoDB 어댑티브 해시 인덱스
  - InnoDB 리두 로그 버퍼

# 4.1.3.2 로컬 메모리 영역

**클라이언트 스레드가 쿼리를 처리하는데 사용하는 메모리 영역**이고, 클라이언트와 MySQL 서버와의 커넥션을 세션이라고 불러 **세션 메모리 영역이라고도 표현**한다.

- 각 클라이언트 스레드 별로 독립적으로 할당되며, 절대 공유되어 사용 X
- 일반적으로 별 신경 쓰지 않고 설정함.
  - 하지만, 낮은 확률로 메모리가 부족해 서버가 다운될 수 있으니, 적절하게 설정하는것이 중요하다.
- 각 쿼리의 용도별로 필요할 때만 공간이 할당, 필요없을 경우 공간을 할당조차도 하지 않을 수 있다.
- **커넥션**이 열려 있는 동안 **계속 할당된 상태로 남는 공간**, 그렇지 않고 **쿼리 실행될 때만 잠깐 할당하는 공간**이 있다.
- 대표적인 로컬 메모리 영역
  - 정렬 버퍼
  - 조인 버퍼
  - 바이너리 로그 캐시
  - 네트워크

# 4.1.4 플러그인 스토리지 엔진 모델

- 사용자 인증 방식, 검색 엔진, 스토리지 엔진 등을 플러그인 형태로 개발해서 사용할 수 있음.
- 사용자가 직접 필요한 스토리지 엔진을 개발해서 사용할 수 있음.
- 데이터 읽기/쓰기만 스토리지 엔진에서 처리.
- 하나의 쿼리 작업은 여러 하위 작업으로 나뉘는데, 각 하위 작업이 MySQL 엔진 영역이랑 스토리지 엔진 영역 중 어느 곳에서 처리하는지 구분할 줄 알아야 한다.

### Handler

- MySQL 엔진이 스토리지 엔진을 조정하기 위해 사용하는 것.
- `Handler_` 로 시작하는 상태변수는 MySQL 엔진이 각 스토리지 엔진에게 보낸 명령의 횟수이다.

# 4.1.5 컴포넌트

- 기존의 플러그인 아키텍처를 대체하기 위해 컴포넌트 아키텍처 지원
- 컴포넌트가 보완해줄 플러그인 단점.
  - 플러그인은 오직 MySQL 서버와 인터페이스할 수 있고, 플러그인끼리 통신할 수 없음.
  - 플러그인은 MySQL 서버의 변수나 함수를 직접 호출하기 때문에 안전하지 않음.(캡슐화 안함)
  - 플러그인은 상호 의존 관계를 설정할 수 없어서 초기화가 어려움.
- 컴포넌트 설치 및 확인
  - 설치 시 추가로 시스템 변수를 설정할 수 있으니, 매뉴얼 확인 필요.

# 4.1.6 쿼리 실행 구조

### 1. 쿼리 파서

- 쿼리 파서는 사용자 요청으로 들어온 쿼리 문장을 토큰으로 분리해 트리 형태의 구조로 만들어 내는 작업을 함.
- 쿼리 문장의 기본 문법 오류는 이 과정에서 발견되고 오류 메시지를 전달.

### 2. 전처리기

- 전 과정에서 만들어진 파서 트리를 기반으로 쿼리 문장에 문제점을 확인.
- 토큰을 테이블 이름, 칼럼 이름, 내장 함수와 같은 객체를 매핑해 객체의 존재 여부와 접근 권한 등을 확인하는 일도 담당.

### 3. 옵티마이저

- 사용자의 요청으로 들어온 쿼리 문장을 저렴한 비용으로 가장 빠르게 처리할지 결정하는 역할을 담당.
- DBMS의 두뇌에 해당한다고 볼 수 있다.
- 이 책에선 옵티마이저가 선택하는 내용을 설명하고, 더 나은 선택을 유도할 방법을 알려줄거임.
- 중요한 역할이고, 영향 범위도 아주 넓다.

### 4. 실행 엔진

- 옵티마이저가 두뇌라면, 실행 엔진과 핸들러는 손과 발에 비유할 수 있음.
- 만들어진 계획대로 각 핸들러에게 요청해서 받은 결과를 또 다른 핸들러 요청의 입력으로 연결하는 역할을 수행.
- Group by 예시
  1. 실행엔진이 핸들러에게 임시 테이블을 만들라고 요청
  2. 다시 실행 엔진은 WHERE 절에 일치하는 레코드를 읽어오라고 핸들러에게 요청
  3. 읽어온 레코드들을 1번에서 준비한 임시 테이블로 저장하라고 다시 핸들러에게 요청
  4. 데이터가 준비된 임시 테이블에서 필요한 방식으로 데이터를 읽어 오라고 핸들러에게 다시 요청
  5. 최종적으로 실행 엔진은 결과를 사용자나 다른 모듈로 넘김

### 5. 핸들러

- MySQL 서버의 가장 밑단에서 실행 엔진의 요청에 따라 데이터를 디스크로 저장, 디스크로부터 읽어오는 역할을 담당.
- 결국 스토리지 엔진을 의미.

# 4.1.8 쿼리 캐시

- 빠른 응답을 필요로 하는 웹 기반의 응용 프로그램에서 매우 중요한 역할을 담당.
- SQL의 실행 결과를 메모리에 캐시하고, 동일 SQL 쿼리가 실행되면 테이블을 읽지 않고 결과 반환. 그래서 빠름.
- 테이블의 데이터가 변경되면 캐시에 저장된 결과 중 변결된 테이블과 관련된 모두를 삭제.
  - 이것 때문에 성능 저하가 심해짐 → MySQL 8.0에 올라오면서 기능이 제거, 시스템 변수도 제거.

# 4.1.9 스레드 풀

- MySQL 엔터프라이즈 에디션에서는 제공되는 기능.
- 이 책은 엔터프라이즈 에디션에 포함된 스레드 풀 대신 Percona Server에서 제공하는 스레드 풀 기능을 살펴본다.
- 내부적으로 사용자의 요청을 처리하는 스레드 개수를 줄여서 동시 처리되는 요청이 많다고 하더라도 MySQL 서버의 CPU가 제한된 개수의 스레드 처리에만 집중할 수 있게 해서 서버의 자원 소모를 줄이는 것이 목적.
  - 성능이 눈에 띄게 향상되는 경우는 드물다.
  - 스케쥴링 과정에서 CPU 시간을 제대로 확보하지 못하는 경우엔 쿼리 처리가 더 느려질 수 있음.
  - 스케쥴링이 잘 된다면 CPU의 프로세서 친화도를 높이고 운영체제 입장에서는 불필요한 Context switch를 줄여, 오버헤드를 낮출 수 있다.
- CPU 코어의 개수만큼 스레드 그룹을 생성.
  - 스레드 구륩 개수는 `thread_pool_size` 를 통해 변경해서 조정 가능.
  - 일반적으로 CPU 코어 개수와 맞추는 것이 CPU 프로세서 친화도를 높이는 데 좋다.
  - 스레드 풀이 처리 중인 작업이 있음에도, 요청이 생기면 `thread_pool_oversubscribe` 시스템 변수에 설정된 개수만큼 추가로 더 받아들여서 처리. 이 값이 너무 크면 스케쥴링해야 항 스레드가 많아져 비효율적으로 작동.
- 스레드 그룹의 모든 스레드가 일을 처리하고 있다면, 새로운 작업 스레드를 추가할지, 기존 작업 스레드가 처리를 완료할 때까지 기다릴지 판단
  - 타이머 스레드가 주기적으로 스레드 그룹의 상태를 체크해서 `thread_pool_stall_limit` 시스템 변수에 정의된 밀리초만큼 작업 스레드가 작업 처리를 못한다면 새로운 작업 스레드를 생성해서 그룹에 추가한다.
    - 응답 시간에 민감한 서비스라면 `thread_pool_stall_limit` 시스템 변수를 적절히 낮춰서 설정.
    - 너무 낮춰 0에 가까운 값으로 설정해야한다면 스레드 풀 기능을 사용하지 않는 편이 나을것이다.
- 선순위 큐와 후순위 큐를 이용해 특정 트랜잭션이나 쿼리를 우선적으로 처리할 수 있는 기능도 제공.
  - 트랜잭션 내에 속한 SQL을 빨리 처리하면 해당 트랜잭션이 가지고 있던 잠금이 빨리 해제되고 잠금 경합을 낮춰 전체적인 처리 성능을 향상

# 4.1.10 트랜잭션 지원 메타데이터

### 메타데이터

데이터베이스 서버에서 테이블의 구조 정보와 스토어드 프로그램 들의 정보를 뜻하고. 데이터 딕셔너리라고 불림.

MySQL 5.7까지는 파일 기반의 메타 데이터 저장으로인해 생성및 변경 작업이 트랜잭션을 지원 X, 비정상 종료시 일관적이지 않은 데이터로 데이터베이스가 깨지는 경우가 발생.

MySQL 8.0 버전부터는 관련 정보를 InnoDB 테이블에 저장. 사용자 인증과 권한 등의 정보를 담는 시스템 테이블과 함께 메타 데이터는 mysql DB에 저장. mysql은 mysql.ibd 테이블 스페이스에 저장.

### InnoDB 스토리지 엔진 외 메타 데이터

InnoDB 스토리지 엔진의 메타 정보는 테이블 기반으로 저장하지만, MyISAM이나 CSV 등과 같은 스토리지 엔진의 메타 정보는 여전히 저장할 공간이 필요.

이를 위해 SDI(Serialized Dictionary Information) 파일을 사용. 기존읜 *.FRM 파일과 동일한 역할을 함. idb2sdi 유틸리티를 이용해 스키마 정보를 추출해 InnoDB도 SDI로 만들 수 있다.

# 4.2 InnoDB 스토리지 엔진 아키텍처

- InnoDB는 MySQL에서 사용할 수 있는 스토리지 엔진 중 거의 유일하게 레코드 기반 잠금을 제공하여 높은 동시성 처리가 가능, 안정적이며 성능이 뛰어남.
  - 하지만 이원화된 잠금 처리탓에 MySQL 명령을 이용해 접근하기 상당히 까다로움.

# 4.2.1 프라이머리 키에 의한 클러스터링

- InnoDB의 모든 테이블은 기본적으로 프라이머리 키를 기준으로 클러스터링 되어 저장된다.
  - 프라이머리 키 값의 순서대로 디스크에 저장된다는 의미.
- 결과적으로 쿼리의 실행 계획에서 프라이머리 키는 기본적으로 다른 보조 인덱스에 비래 비중이 높게 설정.
- MyISAM 스토리지 엔진은 클러스터링 키 지원 안함.
  - 이 때문에 프라이머리 키와 세컨더리 인덱스는 구조적으로 차이가 없다.

# 4.2.2 외래 키 지원

- 이 기능은 InnoDB 스토리지 엔진 레벨에서 지원하는 기능으로 MyISAM, MEMORY 테이블에서는 사용 불가.
- InnoDB에서 FK는 부모, 자식 테이블 모두 해당 칼럼의 인덱스 생성이 필요하고, 변경 시에는 반드시 부모 테이블이나 자식 테이블에 데이터가 있는지 체크하는 작업이 필요해 잠금이 여러 테이블로 전파 → 데드락 발생할 때가 많아짐
- 외래 키가 복잡하게 얽힌 경우에는 `foreign_key_checks` 시스템 변수를 off로 설정해 체크 작업을 진행하면 훨씬 더 빠르게 진행할 수 있다.
  - 외래키 체크를 일시적으로 해제했다고 해서 부모와 자식 테이블 간의 관계가 깨진 상태로 유지해도 된다는 것을 의미하지는 않는다. 해제해도 일관성을 맞춰준 후 다시 외래 키 체크 기능을 활성화 해야한다.

# 4.2.3 MVCC(Multi Version Concurrency Control)

- 일반적으로 레코드 레벨의 트랜잭션을 지원하는 DBMS가 제공하는 기능
- MVCC의 가장 큰 목적은 잠금을 사용하지 않는 일관된 읽기를 제공하는 것이다.
- InnoDB는 언두 로그(Undo log)를 이용해 MVCC를 구현함.
- InnoDB에서는 데이터 수정이 이루어질 때, 새로운 값이 예전 값을 대체하고, 예전 값은 Undo log 영역에 복사가 된다. 이 상태에서 COMMIT 명령을 실행하면 InnoDB는 더 이상의 변경 작업 없이 지금의 상태를 영구적인 데이터로 만들어 버림.
  - 롤백을 실행하면 백업된 데이터를 InnoDB 버퍼 풀로 다시 복구하고,  언두 영역의 내용을 삭제해 버린다.
  - 백업 데이터가 항상 바로 삭제되는 것이 아닌, 언두 영역을 필요로 하는 트랜잭션이 더는 없을 때 비로소 삭제.

# 4.2.4 잠금 없는 일관된 읽기(Non-Locking Consistent Read)

- InnoDB 스토리지 엔진은 MVCC 기술을 이용해 잠금을 걸지 않고 읽기 작업을 수행한다.
  - 잠금을 걸지 않아 읽기 작업은 다른 트랜잭션이 가지고 있는 잠금을 기다리지 않고, 읽기 작업이 가능함.
- 격리 수준이 `SERIALIZABLE` 이 아닌 수준인 경우 INSERT와 연결되지 않은 SELSCT 작업은 다른 트랜잭션의 변경 작업과 관계없이 항상 잠금을 대기하지 않고 바로 실행.
- 레코드를 변경하고 아직 커밋을 수행 안한 경우도 데이터를 읽을 때 조회 작업이 방해 되지 않으며 Undo 영역에서 읽는다. 이것을 잠금 없는 일관된 읽기라 한다.

# 4.2.5 자동 데드락 감지

- InnoDB 스토리지 엔진은 내부적으로 잠금이 교착상태에 빠지지 않았는지 체크하기 위해 잠금 대기 목록을 그래프 형태(Wair-for-List)로 관리한다.
- InnoDB 스토리지 엔진은 데드락 감지 스레드를 가지고 있고, 이 스레드가 주기적으로 잠금 대기 그래프를 검사해 교착 상태에 빠진 트랜잭션들을 찾아서 그중 하나를 강제 종료한다.
  - 강제 종료하는 기준은 언두 로그 양이며, 언두 로그 레코드를 더 적게 가진 트랜잭션이 롤백의 대상이 됨
  - 적게 가진 트랜잭션을 롤백하는 이유는 롤백으로 인한 MySQL 서버의 부하를 덜 유발하기 때문.
- InnoDB 스토리지 엔진은 상위 레이어인 MySQL 엔진에서 관리되는 테이블 잠금은 볼 수 없어 데드락 감지가 불확실할 수 있음.
  - `innodb_table_locks` 시스템 변수를 황성화하면 InnoDB 스토로지 엔진 내부 레코드 잠금과 테이블 레벨의 잠금까지 감지. 특별한 이유 없으면 활성화하자.
- 동시 처리 스레드가 매우 많아지거나 각 트랜잭션이 가진 잠금의 개수가 많아지면 데드락 감지 스레드가 느려짐.
- 느려지는 문제점을 해결하기 위해 `innodb_deadlock_detrct` 를 off로 설정하면 데드락 스레드가 작동 안함.
- 데드락 발생시 중재하는 역할이 없기에 대기하며 `innodb_lock_timeout` 설정 만큼 대기하다가 요청이 실패.

# 4.2.6 자동화된 장애 복구

- InnoDB에는 손실이나 장애로부터 데이터를 보호하기 위한 여러 가지 매커니즘이 탑재돼 있다.
- 그러한 매커니즘을 이용해 MySQL 서버가 시작될 때 일련의 복구 작업이 자동으로 진행된다.
  - InnoDB 데이터 파일은 기본적으로 MySQL 서버가 시작될 때 자동 복구를 수행.
  - 복구 될 수 없는 경우 자동 복구를 멈추고 서버가 종료됨.
    - 복구 될 수 없는 경우는 `innodb_force_recovery` 시스템 변수를 설정해 서버를 시작함.
- 시스템 변수를 설정해도 시작이 되지 않는다면 백업을 통해 다시 구성해야한다.
  - 바이너리 로그를 통해 최대한 최근 시점까지 복구를 시도할 수 있다.

# 4.2.7 InnoDB 버퍼 풀

- InnoDB 스토리지 엔진에서 가장 핵심적인 부분
- 디스크의 데이터 파일이나 인덱스 정보를 메모리에 캐시해 두는 공간
- 쓰기 작업을 지연시켜 일관 작업으로 처리할 수 있게 하는 버퍼 역할도 함.

# 4.2.7.1 버퍼 풀의 크기 설정

- 버퍼 풀의 크기를 동적으로 조절할 수 있음
- 버퍼 풀의 크기를 적절히 작은 값으로 설정해서 조금씩 상황을 봐 가면서 증가시키는 방법이 최적이다.
- `innodb_buffer_pool_size` 시스템 변수로 크기 설정 가능, 가능한 MySQL 서버가 한가한 시점을 골라서 진행.

# 4.2.7.2 버퍼 풀의 구조

- InnoDB 스토리지 엔진은 버퍼 풀이라는 거대한 메모리 공간을 페이지 크기의 조각으로 쪼개, 데이터를 필요로 할 때 해당 데이터 페이지를 읽어서 각 조각에 저장.
- 페이지 조각을 관리하기 위해, LRU(Least Recently Used) 리스트와 플러시(Flush) 리스트, 프리(Free) 리스트 3개의 자료 구조를 사용.
- 처음에 한번 읽힌 데이터 페이지의 사용 빈도가 높으면 그 데이터는 MRU 영역에 있을 확률이 높거나, 영역 내에 있으며, 반대로 사용이 거의 되지 않는다면 LRU의 끝으로 이동하게 되고 InnoDB 버퍼풀에서 제거된다.
- 데이터가 변경되면 리두 로그에 기록 → 버퍼 풀 데이터 페이지에도 변경 내용을 반영

# 4.2.7.3 버퍼 풀과 리두 로그

- InnoDB의 버퍼 풀과 리두 로그는 매우 밀접한 관련을 맺고 있다.
  - 버퍼풀에는 더티 페이지를 가지고 있음.
  - 더티 페이지는 특정 리두 로그 엔트리와 관계를 가지고, 체크포인트가 발생하면 체크포인트 LSN보다 작은 리두로그 엔트리와 더티 페이지 모두 디스크로 동기화돼야한다.

# 4.2.7.4 버퍼 풀 플러시

- 플러시 리스트(Flush_list) 플러시
  - 리두 로그 공간의 재활용을 위해 주기적으로 물갈이 해야한다.
  - 리두 로그 공간이 지워지려면 반드시 더티 페이지가 디스크로 동기화돼야 한다.
  - 동기화를 위해 플러시 리스트 플러시 함수를 호출해서 데이터 페이지를 동기화하는 작업을 수행한다.
- LRU 리스트 플러시
  - InnoDB 스토리지 엔진은 LRU 리스트에서 사용빈도가 낮은 데이터 페이지들을 제거해 새로운 페이지들을 읽어올 공간을 만들어야 하는데, 이 때 LRU 리스트 플러시 함수를 사용함.

# 4.2.7.5 버퍼 풀 상태 백업 및 복구

- MySQL  5.6부터는 버퍼 풀 덤프 및 적재 기능이 도입됐다.
- 서버 점검이나 기타 작업을 위해 MySQL 서버를 재시작 해야 하는 경우 MySQL 서버를 셧다운 하기 전에  `innodb_buffer_pool_dump_now`  시스템 변수를 이용해 InnoDB 버퍼 풀의 상태를 백업할 수 있다.
- 그리고 서버를 다시 시작하면 `innodb_buffer_pool_load_now` 사용해 복구 가능하다

# 4.2.7.6 버퍼 풀의 적재 내용 확인

- MySQL 5.6 버전부터 MySQL 서버의 `information_schema` 데이터베이스의 `innodb_buffer_page` 테이블을 이용해 버퍼풀의 적재 내용을 확인 할 수 있다.
  - 하지만 버퍼 풀이 큰 경우에는 과부화가 걸려 서비스 쿼리가 많이 느려지는 문제가 있다.
- MySQL 8.0 버전에선 문제점을 해결하기 위해  `information_schema` 데이터베이스 안에 `innodb_cached_indexes` 테이블을 추가했다. 이 테이블을 사용하면 테이블의 인덱스 별로 데이터 페이지가 얼마나 적재돼 있는지 확인할 수 있다.

# 4.2.8 Double Write Buffer

- 리두 로그는 공간 낭비를 막기 위해 페이지의 변경된 내용만 기록한다. 이것 때문에 더티 페이지를 디스크 파일로 플러시 할 때, 일부만 기록되는 문제가 발생하면 페이지 내용은 복구할 수 없을 수 있다.
- 이런 문제는  파셜 페이지, 톤 페이지라 부르는 현상을 야기한다. 이 현상은 하드웨어의 오작동, 시스템 셧다운 등으로 발생 할 수 있다.
- 이러한 문제점을 막기 위해 Double-Write 기법을 이용한다.
- 이 기법은 InnoDB 스토리지 엔진이 더티 페이지를 파일에 쓰기 전에 Double Write 버퍼에 등록 시키고, 더티페이지가 정상적으로 기록이 된다면 작동을 안하고 비정상적인 행동을 보이면, 버퍼 내용과 파일의 페이지를 모두 비교해서 다른 페이지를 찾고, 그 페이지를 버퍼에 담긴 페이지로 복사를 하는식으로 작동한다.

# 4.2.9 언두 로그

- DML로 변경되기 이전 버전의 데이터, 즉 백업된 데이터를 말한다.
- 트랜잭션 보장, 격리 수준 보장에 사용된다.

# 4.2.9.1 언두 로그 레코드 모니터링

- 대용량의 데이터를 처리하는 트랜잭션뿐만 아니라 트랜잭션이 오랜 시간 동안 실행될 때도 언두 로그의 양은 급격히 증가할 수 있다. 또한 트랜잭션이 완료됐다고 해서 해당 트랜잭션이 생성한 언두 로그를 즉시 삭제할 수 있는 것은 아니다.
- MySQL 8.0에서는 언두 로그를 돌아가면서 순차적으로 사용해 디스크 공간을 줄이는 것도 가능하며 서버 사용 곤간을 자동으로 줄여 준다. →  이 장점은 언두 로그 공간의 문제점을 해결했다.
- 활성 상태의 트랜잭션이 장시간 유지되는 것은 성능상 안좋다. → 항상 모니터링 해//주는 것이 좋다.

```sql
SHOW ENGINE INNODB STATUS \G

// 8.0 버전에서 사용 가능한 명령
SELECT count
FROM information_schema.innodb_metrics
WHERE SUBSYSTEM='transaction' AND NAME='trx_rseg_history_len';
```

# 4.2.9.2 언두 테이블스페이스 관리

- 언두 로그가 저장되는 공간을 언두 테이블 스페이스라고 한다.
- MySQL 8.0부터 `innodb_undo_tablespaces` 시스템 변수가 효력이 없어졌고, 언두 로그는 항상 시스템 테이블 스페이스 외부의 별도 로그 파일에 기록되도록 개선됐다.
- 언두 테이블스페이스는 1개 이상 128개 이하의 롤백 세그먼트를 가지고, 롤백 세그먼트는 1개 이상의 언두 슬롯을 가진다.
- 언두 로그 슬롯이 부족한 경우 트랜잭션을 시작할 수 없는 심각한 문제를 야기한다. 따라서 언두 로그 관련 시스템 변수를 변경해야 한다면 트랜잭션 개수에 맞게 언두 테이블스페이스와 롤백 세그먼트의 개수를 설정해야한다.
- MySQL 8.0부터 `CREATE UNDO TABLESPACE` 나 `DROP TABLESPACE` 같은 명령으로 새로운 언두 테이블 스페이스를 동적으로 추가하고 삭제할 수 있게 개선. 또한 언두 테이블스페이스 공간을 필요한 만큼만 남기고 불필요하거나 과도하게 할당된 공간을 운영체제로 반납하는 기능도 제공해준다.

# 4.2.10 체인지 버퍼

- RDBMS에서 레코드가 INSERT, UPDATE될 때는 파일을 변경하는 작업뿐 아니라 해당 테이블에 포함된 인덱스를 업데이트하는 작업도 필요하다. 업데이트하는 작업은 랜덤하게 디스크를 읽는 작업이 필요하므로, 인덱스가 많아지면 상당히 많은 자원을 소모하게 된다. 그래서 업데이트를 해야한다면 즉시 실행하지 않고, 임시 공간에 저장해 두고  바로 사용자에게 결과를 반환하는 형태로 성능을 향상 시키는데, 이 때 사용하는 임시 공간이 체인지 버퍼이다.

# 4.2.11 리두 로그 및 로그 버퍼

- 리두 로그는 트랜잭션의 4요소중 하나인 영속성과 가장 밀접하게 연관돼 있다.
- 리두 로그는 비정상적으로 종료됐을 때, 데이터를 잃지 않게 해주는 안전 장치이다.
- 롤백됐지만 데이터 파일에 이미 기록된 데이터는 리두로그로 해결할 수 없다.
- 리두 로그 버퍼링에 사용되는 공간이 로그 버퍼다.

# 4.2.11.1 리두 로그 아카이빙

- MySQL 8.0 버전부터 InnoDB 스토리지 엔진의 리두로그를 아카이빙할 수 있는 기능이 추가됐다.
- 데이터 변경이 너무 많으면, 백업은 실패한다. 하지만, 리두 로그 아카이빙 기능은 변경이 많아 리두 로그가 덮어쓰인다고 하더라도 백업이 실패하지 않게 해준다.

# 4.2.11.2 리두 로그 활성화 및 비활성화

- InnoDB 스토리지 엔진의 리두로그는 어떠한 문제점으로 서버가 비정상적으로 종료됐을 때 데이터 파일에 기록되지 못한 트랜잭션을 복구하기 위해 항상 활성화돼있다.
- MySQL 8.0 이전 버전까지는 수동으로 활성화하거나 비활성화할 수 있게 됐다.
- 대용량 데이터를 한번에 적재하는 경우 비활성화해서 적재 시간을 단축시킬 수 있다.

# 4.2.12 어댑티브 해시 인덱스

- 어댑티브 해시 인덱스는 InnoDB 스토리지 엔진에서 사용자가 자주 요청하는 데이터에 대해 자동으로 생성하는 인덱스이고 특정 시스템 변수를 사용해서 활성화하거나 비활성화할 수 있다.

# 4.2.13 InnoDB와 MyISAM, MEMORY 스토리지 엔진 비교

- 모든 면에서 InnoDB가 좋다.

# 4.3 MyISAM 스토리지 엔진 아키텍처

- MyISAM 스토리지 엔진의 성능에 영향을 미치는 요소는 키 캐시와 운영체제의 캐시/버퍼이다.

# 4.3.1 키 캐시

- InnoDB의 버퍼 풀과 비슷한 역할을 함.
- 이름 그대로 MyISAM 키 캐시는 인덱스만을 대상으로 작동한다. 또한 디스크 쓰기 작업에 대해서만 부분적으로 버퍼링 역할을 한다.

# 4.3.2 운영체제의 캐시 및 버퍼

- 인덱스는 키 캐시를 이용해 디스크를 검색하지 않고도 빠르게 검색할 수 있다.
- MyISAM 스토리지 엔진은 I/O를 해결할 어떠한 캐시나 버퍼링 기능도 가지고 있지 않다.
- 그래서 운영체제의 디스크 읽기 또는 쓰기 작업으로 요청될 수밖에 없다.

# 4.3.3 데이터 파일과 프라이머리 키(인덱스) 구조

- InnoDB 스토리지 엔진을 사용하는 테이블은 프라이머이 키에 의해서 클러스터링되어 저장, MyISAM 테이블은 프라이머리 키에 의한 클러스터링 없이 데이터 파일이 힙 공간처럼 활용된다.
  - 즉, PK와 무관하게 INSERT 순서대로 저장
- MyISAM 테이블에서 ROWID는 가변 길이와 고정 길이의 두 가지 방법으로 저장될 수 있다.

# 4.4 MySQL 로그 파일

- 로그 파일을 잘 알면 서버의 깊은 내부 지식이 없어도 MySQL의 상태나 부하를 일으키는 원인을 쉽게 찾아서 해결할 수 있다.

# 4.4.1 에러 로그 파일

- 에러나 경고 메시지가 출력되는 로그 파일이다.
- .err라는 확장자가 붙은 파일로 생성된다.
  - 시작하는 과정과 관련된 정보성 및 에러 메세지
  - 마지막으로 종료할 때 비정상적으로 종료된 경우 나타나는 InnoDB의 트랜잭션 복구 메시지
  - 쿼리 처리 도중에 발생하는 문제에 대한 에러 메시지
  - 비정상적으로 종료된 커넥션 메시지
  - InnoDB의 모니터링 또는 상태 조회 명령의 결과 메시지
  - MySQL의 종료메시지

# 4.4.2 제너럴 쿼리 로그 파일(제너럴 로그 파일, General log)

- 서버에서 실행되는 쿼리로 어떤 것들이 있는지 전체 목록을 뽑아서 검토해 볼 때가 있는데, 이 때 쿼리 로그를 활성화해서 쿼리를 쿼리 로그로 기록 → 기록한 파일 검토

# 4.4.3 슬로우 쿼리 로그

- MySQL 서버의 전체적인 성능 저하를 검사하거나 정기적인 점검을 위한 튜닝할 때, 어떤 쿼리가 문제인지 판단하기 어렵다. 이럴 때 슬로우 쿼리 로그가 상당히 도움이 된다.

일반적으로 슬로우 쿼리 또는 제너럴 로그 파일의 내용이 상당히 많아서 직접 쿼리를 검토하기 시간이 많이 걸리거나 어느 쿼리를 집중적으로 튜닝해야 할지 식별하기가 어려울 수 있다.

→ 이런 경우는 Percona에서 개발한 Percona Toolkit의 `pt-query-digest` 스크립트를 이용하면 된다.
